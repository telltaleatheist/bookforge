# BookForge TTS API — Browser Extension Integration Guide

This document is a complete, self-contained spec for building a **Chrome extension** that streams text-to-speech audio from BookForge. The server side is already built and tested; the extension is a separate program and needs nothing from the BookForge codebase except what's written here.

## What the extension does (product intent)

- Detects text blocks (paragraphs, article sections) on any webpage and shows a small **play button** next to each block.
- Clicking play shows a compact **play / pause / rewind / fast-forward** control set, loads audio from the BookForge TTS server, and starts playing as soon as the first audio arrives (~2–3 s when the engine is warm).
- Pause/seek/rewind are **entirely local**: the extension keeps all received audio, so transport controls never re-contact the server. The only server verbs are "speak this text" and "cancel".
- Runs against a local BookForge install today, but the protocol and auth are LAN-ready (the server can be configured to listen on the network), so a "connect to another machine" setting is a natural future addition.

## The server

BookForge (an Electron desktop app) runs a WebSocket server that fronts a pool of XTTS (neural TTS) worker processes (worker count is user-configurable, default 4 on CPU).

| Property | Value |
|---|---|
| URL | `ws://127.0.0.1:8766` (port/host configurable in BookForge) |
| Transport | WebSocket, JSON text frames only |
| Auth | Shared token, sent in the first message (see below) |
| Detection probe | Plain HTTP `GET http://127.0.0.1:8766/` returns `{"service":"bookforge-tts","version":1}` |
| Audio format | Base64-encoded PCM16 (signed 16-bit little-endian), **24,000 Hz, mono** |
| Protocol version | 1 (echoed in the `hello` reply) |

The server only runs while the BookForge app is open. The HTTP probe is the cheap way to show "BookForge not running" vs "connected" in the extension UI.

### Token

The token is auto-generated by BookForge into:

```
~/Library/Application Support/bookforge-app/tts-api.json     (macOS)
%APPDATA%/bookforge-app/tts-api.json                          (Windows)
```

File shape: `{"port": 8766, "host": "127.0.0.1", "token": "<base64url string>"}`.

The easiest way for the user to get it is **BookForge → Settings → TTS API**, which shows the token with a Copy button (plus the port and a LAN-access toggle).

The extension should have an options page where the user pastes the token (and optionally a host/port for the LAN case). Defaults: host `127.0.0.1`, port `8766`. Store in `chrome.storage.local`.

A wrong or missing token gets the socket closed with code **4401**. The server also closes 4401 if any action is sent before a successful `hello`, or if `hello` doesn't arrive within 10 seconds of connecting.

## Protocol

All frames are JSON objects. Client messages carry an `action`; server messages carry a `type`.

### Client → server

```jsonc
{"action": "hello", "token": "<token>"}            // REQUIRED first message

{"action": "status"}                               // poll engine status

{"action": "engine.start", "voice": "RayPorter"}   // optionally pre-warm the engine (voice optional)
{"action": "engine.stop"}                          // shut the engine down (frees ~20 GB RAM)

{"action": "speak",                                // the main verb
 "requestId": "blk-17-1718120000",                 // client-chosen string or number; echo key for all events
 "text": "Raw text of the block. Multiple sentences are fine.\n\nParagraphs too.",
 "settings": {                                     // optional, all fields optional
   "voice": "ScarlettJohansson",                   // omit to use the engine's current/last voice
   "speed": 1.1,                                   // playback-rate baked into the audio, default 1.0
   "temperature": 0.65, "topP": 0.85, "repetitionPenalty": 2.0   // sampling knobs, rarely needed
 }}

{"action": "playhead", "requestId": "blk-17-…", "sentenceIndex": 4}   // see "Lookahead" below

{"action": "cancel", "requestId": "blk-17-…"}      // stop generating (user closed controls / navigated away)
```

### Server → client

```jsonc
{"type": "hello", "version": 1, "state": "stopped|starting|running",
 "serviceMode": false, "voices": ["ScarlettJohansson", "..."], "currentVoice": null}
// Reply to hello. voices is [] while the engine is cold (it's discovered on engine start).

{"type": "status", ...same fields as hello minus version}   // reply to status / engine.start

{"type": "state", "state": "stopped|starting|running", "serviceMode": bool}
// PUSHED to all connected clients whenever engine state changes. Use it to drive
// a "warming up…" spinner: speak on a cold engine triggers starting → running.

{"type": "speaking", "requestId": "...", "sentences": ["First sentence.", "Second one."]}
// The server segmented your text. Arrives before any audio. Array order = sentenceIndex order.

{"type": "chunk", "requestId": "...", "sentenceIndex": 0, "seq": 0,
 "data": "<base64 pcm16>", "duration": 0.82, "sampleRate": 24000}
// One piece of audio. See "Audio assembly" for ordering rules.

{"type": "done", "requestId": "...", "sentenceIndex": 0, "duration": 2.5}
// This sentence is fully generated (all its chunks have been sent).

{"type": "failed", "requestId": "...", "sentenceIndex": 3, "error": "..."}
// This sentence failed; others continue. Skip it during playback.

{"type": "complete", "requestId": "..."}
// Every sentence has been generated. No more audio for this request.

{"type": "cancelled", "requestId": "..."}
// The request was stopped — either you sent cancel, or something else preempted
// the engine (another speak, or the user pressing play inside BookForge itself).
// Whatever audio you already received remains playable.

{"type": "error", "requestId": "...", "message": "..."}   // requestId present when tied to a speak
```

### Speak lifecycle (happy path)

```
→ speak {requestId, text}
← state {starting}            (only if the engine was cold; takes ~60s, voices load)
← state {running}
← speaking {sentences: [...]}  (text segmented; you now know the sentence count)
← chunk {sentenceIndex:0, seq:0}   ← first audio, START PLAYBACK HERE
← chunk {sentenceIndex:0, seq:1}
← done  {sentenceIndex:0}
← chunk {sentenceIndex:2, seq:0}   ← note: sentence 2 before 1 — see below
← chunk {sentenceIndex:1, seq:0}
← done  {sentenceIndex:1} … 
← complete
```

### Audio assembly rules (important)

Multiple workers generate in parallel (4 by default), so **sentences complete out of order**:

- Sentence **0** (the playhead sentence) streams in multiple small chunks (`seq` 0,1,2,… ~1 s of audio each) so playback can start fast. Chunks *within* one sentence always arrive in `seq` order.
- All other sentences arrive as a **single chunk** (`seq: 0`) each, in whatever order workers finish.
- Buffer per `(sentenceIndex, seq)`. A sentence is playable once its `done` event arrives (or, for sentence 0, you can start playing chunks as they arrive). Play sentences strictly in `sentenceIndex` order; if the next sentence isn't done yet, wait (show buffering) — in practice generation runs ~2× realtime so this is rare after the start.
- Concatenated PCM of all sentences in index order = the full audio for the block. There is no silence padding between sentences; small natural pauses are already in the generated audio.

Decoding a chunk: `atob` → `Uint8Array` → `Int16Array` → `Float32Array` (divide by 32768) → `AudioBuffer` at 24,000 Hz, 1 channel. Schedule gaplessly with `AudioBufferSourceNode.start(when)` using an accumulating time cursor on one `AudioContext`.

### Lookahead and the `playhead` action

The server throttles generation to **~45 seconds of audio ahead of the reported playhead** (so pausing doesn't burn GPU on audio nobody may hear). Consequences:

- Blocks shorter than ~45 s of speech: everything generates immediately; you can ignore `playhead` entirely.
- Longer blocks: generation will idle at the 45 s window until you advance it. Send `{"action":"playhead","requestId","sentenceIndex":N}` whenever playback crosses into sentence N (cheap; once per sentence boundary is right). If playback is paused, simply don't send it — generation pauses with you and resumes when you do.
- The playhead only moves **forward** on the server (backward reports are ignored). That's fine: rewind plays from the local buffer and never needs the server.

### Concurrency model

There is **one generation session globally**. A new `speak` — from any extension tab, any client, or BookForge's own Listen window — preempts the current one, which receives `{"type":"cancelled"}`. This is the desired UX (clicking play on block B while block A is loading should switch to B), but build the extension so a `cancelled` event cleanly finalizes whatever audio that request already has rather than treating it as an error.

Multiple WebSocket connections are fine; audio events are routed only to the connection that sent the `speak`.

### Engine lifecycle notes

- `speak` on a cold engine **auto-starts it** (N Python processes — 4 by default, configurable in BookForge → Settings → TTS Server — ~5 GB RAM each, ~60 s to ready on the user's M1 Ultra). Show "starting TTS engine…" while `state` is `starting`. With fewer workers, generation is slower than the default ~2× realtime, so buffering pauses mid-playback are more likely — the extension shouldn't assume generation outruns playback.
- The engine idles out after **10 minutes** without generation (unless the user pinned it as a service inside BookForge). A `speak` after idle-out just cold-starts again — handle it with the same spinner.
- Voice switching is valid per-`speak` via `settings.voice` but reloads models on all workers (takes a while); the extension should default to *omitting* `voice` so it uses whatever is already loaded, and offer voice choice in options. Voice names come from the `hello`/`status` reply (e.g. `ScarlettJohansson`, `DavidAttenborough`, `MorganFreeman`, `NeilGaiman`, `RayPorter`, `RosamundPike` — but always use the list from the server, it's discovered at runtime).

## Chrome MV3 architecture (recommended)

MV3 service workers can't own an `AudioContext` and get killed after ~30 s idle, so:

```
content script (per page)
  - finds text blocks (e.g. <p>, <article> children; min text length threshold)
  - injects play buttons + the transport control UI
  - extracts block text (innerText, normalized whitespace)
  - messages the service worker: {cmd:'speak', text, blockId}
        │ chrome.runtime messaging
service worker (background)
  - creates/ensures the OFFSCREEN DOCUMENT (chrome.offscreen API,
    reason: AUDIO_PLAYBACK, justification: "TTS audio playback")
  - relays commands/status between content script and offscreen doc
        │ chrome.runtime messaging
offscreen document  ←— owns ALL of:
  - the WebSocket connection (persistent, reconnect with backoff)
  - the protocol state machine (hello/auth, speak requests, chunk buffering)
  - AudioContext + gapless scheduling + pause/seek/rate
  - reports playback position + state back up for the page UI
```

- One offscreen document serves all tabs; one audio stream plays at a time (matches the single-session server model).
- Manifest needs `"permissions": ["offscreen", "storage"]`. A WebSocket from an extension context to `ws://127.0.0.1` does not need host permissions, but add `"host_permissions": ["http://127.0.0.1/*"]` if you use the HTTP probe via `fetch`.
- Pause = `AudioContext.suspend()` (or stop sources and remember offset). Rewind/FF = recompute which sentence/offset the target time falls in from the per-sentence durations (`done` events carry exact durations) and reschedule from there — all buffered locally, no server round-trip.
- Track per-request state keyed by `requestId`; generate IDs like `${blockId}-${Date.now()}`.

## Reference client

A working Node.js client lives in the BookForge repo at `scripts/tts-api-client-example.mjs` — it connects, authenticates, speaks a text block, assembles the chunks, and writes a WAV. The browser version differs only in using `new WebSocket(url)` and Web Audio instead of file output. Quick gist:

```js
const ws = new WebSocket('ws://127.0.0.1:8766');
ws.onopen = () => ws.send(JSON.stringify({ action: 'hello', token }));
ws.onmessage = (e) => {
  const msg = JSON.parse(e.data);
  if (msg.type === 'hello')   ws.send(JSON.stringify({ action: 'speak', requestId: 'r1', text }));
  if (msg.type === 'chunk')   bufferChunk(msg);          // and start playback on the first one
  if (msg.type === 'done')    markSentencePlayable(msg);
  if (msg.type === 'complete') allAudioReceived();
};
```

## Error handling checklist

- HTTP probe fails / WS won't connect → "BookForge isn't running".
- Close code 4401 → "Check your token in extension options".
- `{"type":"error"}` with a `requestId` → that speak failed (engine failed to start, empty text, etc.); show the message, reset that block's button.
- `failed` for one sentence → skip it in playback, keep going.
- `cancelled` → finalize the request with whatever audio arrived; if it was preempted by the user's own new play click, that's expected flow.
- Socket drops mid-speak → reconnect, re-`hello`; the old request is gone (the server cancels generation when a client disconnects). Re-`speak` from the current sentence's text onward if the user resumes... or simply replay from local buffer if everything needed had already arrived.
